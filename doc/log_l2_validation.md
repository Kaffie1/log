# L2 原始数据日志验证方案

## 1. 目的

本文档用于定义 `L2` 原始数据日志在实现完成后的验证范围、测试分类与测试用例。

本次验证重点回答以下问题：

1. 原始数据日志是否能够稳定录制且不丢失数据
2. 分段是否会在达到指定大小后自动触发，且分段前后记录连续
3. 正常退出时是否能够先完成封口，且最后一段数据完整
4. 异常退出后下次启动是否能够恢复未封口文件，且恢复后记录无缺口
5. 后台处理与前台录制职责边界是否符合预期，且处理前后数据一致
6. 旧遗留目录和历史兼容场景是否处理正确

## 2. 验证范围

本次验证覆盖以下内容：

1. 预定义 topic 的接入与原始数据落盘
2. `business_data`、`static_data`、`large_data` 的目录分类
3. `.data`、`.idx`、`.meta` 文件组织与命名规则
4. 按大小自动分段
5. 正常退出封口
6. 启动前未封口恢复
7. 后台压缩处理协同
8. 旧遗留目录清理
9. 长时间运行与异常场景稳定性

本次验证不以吞吐量、实时性和峰值能力为主要目标，`L2` 的核心验收标准是原始数据不丢失、文件边界正确、异常后可恢复完整性。

## 3. 测试分类

### 3.1 基础功能测试

目标：

1. 验证 `L2` 能够正常启动、接收数据并完整落盘
2. 验证不同 topic 能够进入正确的分类目录
3. 验证无数据 topic 不会产生无效文件

### 3.2 分段行为测试

目标：

1. 验证分段由文件大小阈值自然触发
2. 验证分段不会依赖手工强制切段
3. 验证多 topic 并发时各自独立分段
4. 验证分段后的命名、边界和记录连续性正确

### 3.3 生命周期测试

目标：

1. 验证正常退出时当前活动文件能够先封口且末尾数据完整
2. 验证封口后文件能够进入后续处理链路且处理前后数据一致
3. 验证运行中活动文件不会被提前错误处理

### 3.4 异常与恢复测试

目标：

1. 验证异常断电或异常退出后的恢复能力
2. 验证启动前未封口文件能够完成恢复封口且记录连续
3. 验证异常文件不会影响其他正常文件继续录制

### 3.5 后治理协同测试

目标：

1. 验证前台录制、封口和后台处理边界清晰
2. 验证已封口文件能够被后台识别
3. 验证已处理文件不会被重复处理且不会引入数据变化

### 3.6 健壮性测试

目标：

1. 验证长时间运行的稳定性
2. 验证磁盘异常、目录权限异常等失败场景
3. 验证系统在异常条件下的可观测性与可恢复性

## 4. 测试环境建议

建议测试环境满足以下条件：

1. 使用与目标部署一致的 `Docker` 或目标运行环境
2. 使用稳定的测试 topic 发布工具持续发包
3. 可调整单 topic 分段阈值，便于观察自然分段
4. 可访问日志根目录，便于检查输出文件和目录状态
5. 可模拟正常退出、异常杀进程和重启恢复场景

## 5. 测试用例表

| 编号 | 功能点 | 前置条件 | 测试步骤 | 预期结果 |
| --- | --- | --- | --- | --- |
| TC01 | 基本录制 | 录制器正常启动，topic 正常发布 | 启动 recorder，启动测试 topic 发布，持续运行一段时间 | 生成对应 topic 的日志文件，文件名和目录层级符合设计，记录条数与输入一致 |
| TC02 | 多 topic 同时录制 | 配置中包含多类 topic | 同时发布多类 topic 数据 | 不同 topic 按各自目录落盘，互不干扰，各 topic 记录条数分别与输入一致 |
| TC03 | 空闲 topic 不生成文件 | 某些配置 topic 无数据输入 | 启动 recorder，但只发部分 topic | 未收到数据的 topic 不应生成无效日志文件 |
| TC04 | 按大小自动分段 | 分段阈值设为可观测的小值 | 连续发包直到超过阈值，不做任何手工切段 | 文件达到阈值后自动生成下一段，旧段结束，新段继续写入，分段前后总记录数连续无缺口 |
| TC05 | 分段阈值边界 | 阈值可控，消息大小可估算 | 写入累计大小接近阈值后再追加一条消息 | 分段在越过阈值后触发，不提前、不漏切，边界附近记录无丢失无重复 |
| TC06 | 高频写入下分段 | 高频率持续发包 | 在高频场景下连续运行 | 仍能稳定按阈值分段，无卡死、无丢段，分段后记录顺序连续 |
| TC07 | 多 topic 并发分段 | 多个 topic 同时快速写入 | 同时给多个 topic 持续发大包 | 各 topic 独立判断和分段，不会互相串文件，各自记录集合完整 |
| TC08 | 分段后文件命名连续性 | 已触发多次自动分段 | 检查同一 topic 的多段文件命名 | 每段命名合法、时间范围合理、无重复覆盖，跨段时间边界与记录范围一致 |
| TC09 | 正常退出封口 | recorder 已写入数据 | 发送正常 shutdown 或 stop | 退出前当前活动文件完成封口，不留活动态文件，最后一段记录完整可读 |
| TC10 | 正常退出后后治理协同 | 已产生已封口文件 | 正常退出 recorder，等待后台处理完成 | 封口后文件进入后治理流程，目录状态符合预期，处理前后记录条数和内容一致 |
| TC11 | 退出时仅处理本进程文件 | 同机存在其他无关日志目录 | 关闭当前 L2 recorder | 只处理当前 recorder 关联文件，不误处理其他无关数据 |
| TC12 | 异常退出恢复 | 运行中存在活动文件 | 直接杀进程或模拟断电，下次重新启动 | 启动前发现未封口活动文件，并先完成封口恢复，恢复后记录范围连续无缺口 |
| TC13 | 恢复时不误改历史文件 | 目录中同时存在历史正常段和异常活动段 | 重启 recorder | 仅未封口活动文件被恢复封口，历史正常文件不被重复处理，历史数据保持不变 |
| TC14 | 恢复后继续录制 | 上次异常退出留下活动文件 | 重启后继续正常发包 | 恢复完成后可继续录制，新旧数据边界清晰，恢复前后总记录集合完整 |
| TC15 | 多组遗留文件恢复 | 目录中存在多个 topic 的未封口文件 | 启动 recorder | 所有遗留活动文件都能被识别并完成封口，各文件恢复后记录完整 |
| TC16 | 数据文件与索引文件一致恢复 | 人工构造成对遗留文件 | 启动恢复 | 成对文件按同一规则恢复，结果一致，索引与数据记录范围一致 |
| TC17 | 残缺文件异常处理 | 人工构造缺失 data 或 idx 的残缺文件 | 启动恢复 | 系统能识别异常并给出可观测结果，不影响其他正常文件恢复 |
| TC18 | 后治理结果校验 | 已有封口文件待处理 | 等待后治理处理完成 | 目标文件完成预期处理，结果文件可识别、可读取，处理不改变数据内容 |
| TC19 | 处理前后数据一致性 | 已知消息条数和时间范围 | 录制、封口、后治理后再核对结果 | 数据条数、时间范围和主题内容保持一致，无缺失、无重复、无越界 |
| TC20 | 活动文件不提前处理 | 运行中存在活动文件 | 在未 shutdown 前观察目录 | 活动文件不应被错误当成已完成文件处理 |
| TC21 | 已处理文件不重复处理 | 某批文件已完成处理 | 再次触发后台扫描 | 不应重复处理，不应生成重复产物 |
| TC22 | 旧遗留目录清理 | 目录中残留旧 `shutdown bundle` 产物 | 启动或运行新版本 | 旧遗留目录被清理，运行过程中不再生成新的旧目录 |
| TC23 | 启动恢复接口有效性 | 启动前存在活动文件 | 按当前恢复链路启动 | 启动恢复流程能够完成统一封口 |
| TC24 | 正常退出顺序正确 | 正在录制且存在活动文件 | 触发正常退出 | 顺序满足先停止写入、再封口、再进入后治理，收尾过程中不引入末尾数据缺失 |
| TC25 | 空目录启动 | 日志目录为空 | 启动 recorder，不发数据，再退出 | 启停正常，不报错，不生成无意义文件 |
| TC26 | 单条大消息边界 | 分段阈值较小 | 发送单条超阈值消息 | 系统行为符合设计，不崩溃，不影响后续写入 |
| TC27 | 长时间运行稳定性 | 正常部署环境 | 长时间持续发包运行 | 持续分段稳定，无明显泄漏，无目录异常增长，长稳过程中记录累计无异常缺口 |
| TC28 | 目录权限或磁盘异常 | 人工制造不可写目录或磁盘不足 | 启动录制或运行中写入 | 错误可观测，系统行为符合容错预期，不产生不可控脏状态 |

## 6. 重点验收项

以下内容应作为本轮验证的重点验收项：

1. 原始数据不丢失是 `L2` 的首要验收标准
2. 分段必须由达到配置大小阈值自然触发，而不是依赖手工强制切段
3. 分段前后、恢复前后、处理前后都必须保证记录连续，无缺失、无重复、无越界
4. 正常退出时当前活动文件必须先封口，再进入后续处理
5. 异常退出后，下次启动前必须先完成未封口文件恢复
6. 已处理文件不能重复处理，活动文件不能被提前处理
7. 旧 `shutdown bundle` 目录不应再生成，历史遗留应可清理

## 7. 数据完整性检查原则

关键测试项应统一按照以下原则检查是否通过：

1. 发送端消息条数与落盘后可解析消息条数一致
2. 同一 topic 的消息序号或等价顺序标识连续，无异常跳变
3. 分段前后总记录集合与发送集合一致，不因切段产生缺口或重复
4. 异常恢复前后文件时间范围和记录范围连续
5. 后治理处理前后数据内容、时间范围和条数保持一致

## 8. 建议执行顺序

建议按以下顺序组织验证：

1. 先执行 `TC01` 至 `TC03`，确认基础录制行为正确
2. 再执行 `TC04` 至 `TC08`，重点验证按大小自动分段
3. 再执行 `TC09` 至 `TC11`，确认正常退出封口与后治理协同
4. 再执行 `TC12` 至 `TC17`，确认异常退出恢复能力
5. 最后执行 `TC18` 至 `TC28`，补齐后治理、遗留清理与健壮性验证
