# 背景

当前本体中没有统一的日志系统，需要定义一个日志框架。当前本体系统分为平台层，应用层，ROS中间件，算法，嵌入式。当前没有自动清理等功能，导致出现一些问题，需要一个统一的日志系统来处理通用性的能力。本文档定义了日志系统需要实现的能力，用于对后续实现日志系统进行参考。

# 业务价值

1. 提高排查效率：统一查询入口，按照trace_id进行追踪
2. 保障系统稳定性：自动清理+轮转，永不写满磁盘
3. 保障实时性：日志写入不阻塞控制线程

# 本期范围

1. 本体的日志系统
2. 统一日志格式定义
3. 自动轮转+自动清理
4. 基础查询能力
5. 动态调级能力
6. trace_id跨层追踪

# 需求详情

用户故事：作为开发，我希望所有层的日志格式统一，这样不用为每一个层级写不同的解析脚本。

## 统一字段定义

| 字段                 | 类型   | 必填 | 说明                                                   |
| -------------------- | ------ | ---- | ------------------------------------------------------ |
| version              | string | 是   | 格式版本，固定"1.0"                                    |
| timestamp_us         | int64  | 是   | 事件发生时间（微秒），由SDK生成                        |
| receive_timestamp_us | int64  | 否   | 日志进程接收时间（微秒），用于诊断网络延迟             |
| robot_sn             | string | 是   | 机器人序列号                                           |
| layer                | string | 是   | 层级：platform/app/ros/algorithm/embedded              |
| module               | string | 是   | 模块名，例如：uplimb                                   |
| level                | string | 是   | DEBUG/INFO/WARN/ERROR/FATAL                            |
| trace_id             | string | 否   | 全链路追踪ID，由入口处生成                             |
| file                 | string | 否   | 源文件名 (仅DEBUG级别输出)                             |
| line                 | int    | 否   | 行号 (仅DEBUG级别输出)                                 |
| func                 | string | 否   | 函数名 (仅DEBUG级别输出)                               |
| message              | string | 是   | 日志内容，仅存放静态文本模板，严禁放入动态变量         |
| extra                | object | 否   | 扩展字段，所有动态数值，状态必须放入此处，便于后续统计 |

## 日志级别定义

| 级别  | 含义                                              | 标准                                                         | 使用场景               | 生产环境是否开启 |
| ----- | ------------------------------------------------- | ------------------------------------------------------------ | ---------------------- | ---------------- |
| DEBUG | 开发调试用                                        | 用于复现和定位具体代码逻辑问题的详细数据。： 1. 循环内的详细数据（如：每帧的坐标、每秒的循环计数）。 2. 函数入口/出口参数。 | 开发调试、详细执行流程 | 关闭             |
| INFO  | 系统正常运行的里程碑，用于回答“系统现在在干嘛”。  | 记录系统关键状态变更和阶段性成果，用于排查业务流程断点。： 1. 生命周期事件（启动、停止、初始化完成）。 2. 状态机切换：系统或者业务状态机切换，资源管理器分配日志 3. 关键配置加载成功。 | 关键业务流程节点       | 开启             |
| WARN  | 当前业务能跑，但处于“亚健康”状态，未来可能变Error | 不影响核心流程，但提示系统存在隐患或正在进行非预期的降级处理。： 参数越界（如：电池温度偏高但未超限、使用了默认配置启动）。 重试行为（如：网络请求超时正在重试）。 非关键功能降级（如：语音播报失败，但导航正常）。 | 异常但可自动恢复       | 开启             |
| ERROR | 当前任务失败，但进程尚存，系统处于不稳定状态      | 影响当前任务执行，需要运维关注或自动触发恢复流程的错误。： 1. 外部依赖失败（如：连接云端失败、数据库连不上）。 2. 硬件响应异常（如：电机通讯超时、传感器数据校验错误）。 3. 关键算法失败（如：路径规划无解、定位丢失）。 | 功能失败但系统可运行   | 开启             |
| FATAL | 进程无法继续运行，必须重启或人工介入。            | 任何导致模块无法启动或必须强制退出的错误： 1. 核心进程崩溃。 2. 关键硬件不可用（如：找不到IMU设备文件）。 3. 系统资源耗尽（如：磁盘已满无法写入、内存耗尽）。 | 进程即将退出           | 开启             |

# 日志缓冲

1. 日志需要采用非阻塞写入，不影响业务操作
2. 日志队列满时，warn以下可以丢弃，以上不允许丢弃
3. 日志队列容量由开发根据实际情况定义

## 其他要求

# 日志存储

## 存储路径

```Plain
/var/log/robot/
├── {module1}/
│   ├── {module1}.log.{YYYYMMDD}      # 当前活跃文件
│   ├── {module1}.log.{YYYYMMDD}_1.gz # 轮转压缩文件
│   └── ...
├── {module2}/
└── system/
    ├── logd.log
    └── time_sync.log  # 时间同步日志
```

## 轮转策略

1. 单文件达到阈值需要进行轮转，默认值50MB
2. 轮转后需要进行压缩，压缩算法开发自行定义
3. 轮转后需要在新日志中保存最后100条日志，即轮转时将旧文件最后100条复制到新文件开头

## 自动清理策略

1. 日志总容量超过上限自动删除最旧日志，默认2GB（配置项，根据实际情况调整）
2. 保留最近N天的日志，默认30天
3. 系统每10分钟执行一次清理检查
4. 当系统磁盘小于500MB时，只写warn以及fatal级别日志
5. 当磁盘小于100MB时，删除存储的最早的日志，直到大于100MB

# 日志查询

1. 需要支持如下能力，使用命令行的形式
   1. 实时查看日志：需要支持实时跟随，指定模块，指定级别，指定日志范围搜索，显示行数，查询热数据（最近100条），查看各模块日志大小和条数
   2. 搜索日志：需要支持指定日志等级和模块
   3. 统计信息：当前日志占用总容量的百分比，日志条数
   4. 手动清理日志：可以指定等级，指定时间（之前的删除）
   5. 支持按照traceID查询调用链路

# 日志等级调整

1. 支持运行时调整指定模块的日志级别
2. 支持修改立即生效，无需重启进程

# 链路追踪

1. 各层日志需要携带traceID
2. traceID通过请求头以及ROS内容透传
3. 日志查询需要支持按照traceID查询完整调用链路

# 日志采样

1. 支持日志采样接口
2. 支持采样间隔配置，按秒
3. 采样日志标记为sample，可以过滤

# 各层接入需求

1. 平台，应用和算法层需要统一使用SDK
2. ROS中间件需要提供日志转发节点

# 日志系统配置管理

日志系统本身的参数需要以yaml格式进行定义，如下是需要包含的配置字段：

静态配置：

1. 日志根目录
2. 日志系统本身的日志级别（默认info）
3. 单文件最大大小（默认50MB）
4. 日志文件保留时间（默认30天）
5. 压缩阈值：当磁盘空间大于此值时进行压缩，否则直接删除旧日志。需要先进行压缩，如果压缩后还是大于，则进行删除
6. 磁盘告警阈值：当磁盘空间低于此值时，记录日志，默认500mb
7. 磁盘紧急阈值：当磁盘空间低于此值时，只记录error和fatal级别日志，默认50mb
8. 批量写入大小：每次从从日志队列写多少条写入磁盘，默认100
9. 写入间隔：没有到批量写入大小也要强制写入：默认1000毫秒
10. 日志队列默认大小：默认20000条

# 日志系统本身的监控

1. 队列积压场景：当日志队列积压超过80%时，需要记录一条warn级别的日志告警
2. 丢弃统计：每60秒打印一次“本分钟丢弃日志数”

# 日志加密需求

日志采用对称加密，实现方式以及算法开发可自行定义

# 非功能性约束

1. SDK写入日志耗时必须＜10us，禁止阻塞业务控制线程
2. CPU占用率需要小于1%
3. 占用内存上限控制在50MB
4. 日志进程奔溃后，需要在1S内自动拉起
5. 当日志进程异常退出时，最后几条日志允许丢失，但不能导致文件损坏无法读取
6. 当日志文件损坏时，需要自动备份损坏文件并新建文件，而不是启动失败
7. 日志系统本身级别为info，本身的轮转和清理策略与普通日志一致
