# L3 日志验证与评估方案

## 1. 目的

本文档用于定义 `L3` 业务日志在实现完成后的验证、测试与评估方案。

验证重点不再是“接口是否存在”，而是回答以下问题：

1. 功能是否正确
2. 异常场景下是否可恢复
3. 是否满足实时性要求
4. 是否有效降低磁盘 `IO`
5. 吞吐量上限和资源占用是否满足预期
6. 与开源日志方案相比是否具备足够优势

## 2. 验证范围

本次验证覆盖以下内容：

1. `C++` 接口与 `Python` 接口行为一致性
2. 日志写入、轮转、压缩、清理、刷新、关闭
3. 异常退出、磁盘满、压缩失败、文件损坏等失败策略
4. 实时性、吞吐量、磁盘 `IO`、CPU、内存等性能指标
5. 与 `spdlog`、`ROS` 常见日志方案的对比评估

## 3. 测试分类

### 3.1 功能测试

目标：

1. 验证 `Init/Flush/Shutdown` 行为正确
2. 验证 `LOG_TRACE/DEBUG/INFO/WARN/ERROR/CRITICAL` 输出正确
3. 验证模块名、级别、时间戳、payload 等字段写入正确
4. 验证 `C++` 与 `Python` 输出格式一致

建议测试项：

1. 单模块单线程写入
2. 单模块多线程写入
3. 多模块并发写入
4. 不同级别过滤结果
5. `Flush` 后文件可读性
6. `Shutdown` 后文件完整性

### 3.2 生命周期测试

目标：

1. 验证文件轮转是否按阈值触发
2. 验证轮转后旧文件是否正确压缩
3. 验证清理策略是否按保留时间和数量生效
4. 验证新文件是否保留上一文件尾部必要内容

建议测试项：

1. 小阈值下快速触发轮转
2. 连续轮转后的文件命名检查
3. 压缩成功与压缩失败分支
4. 清理窗口边界测试
5. 长时间运行后的目录稳定性检查

### 3.3 异常与恢复测试

目标：

1. 验证失败策略是否真实可用
2. 验证异常场景下不会导致日志系统彻底不可用
3. 验证故障后是否能够自动恢复或安全降级

建议测试项：

1. 日志目录不存在
2. 日志目录无写权限
3. 磁盘空间不足
4. 磁盘只读
5. 压缩线程异常
6. 文件写入中断
7. 进程异常退出
8. 文件损坏后重新启动
9. 队列满载
10. 后台线程阻塞或崩溃

重点观察项：

1. 是否丢日志
2. 是否阻塞业务线程
3. 是否产生损坏文件
4. 是否进入预期降级模式
5. 是否能重新恢复写入

### 3.4 实时性测试

目标：

1. 验证日志调用对业务线程影响足够小
2. 验证高频场景下不会引入明显阻塞

核心指标：

1. 单次日志调用耗时
2. `P50/P95/P99` 延迟
3. 最大阻塞时间
4. 不同 payload 大小时的延迟变化

建议测试场景：

1. 单线程高频 `LOG_INFO`
2. 多线程并发高频写入
3. 不同日志级别混合写入
4. 压缩开启与关闭下的延迟对比

### 3.5 吞吐与磁盘 IO 测试

目标：

1. 验证系统最大吞吐能力
2. 验证批量写入和异步队列是否有效降低磁盘 `IO`
3. 验证不同参数组合对性能的影响

核心指标：

1. 每秒写入条数
2. 每秒写入字节数
3. 磁盘写入带宽
4. 磁盘 `IOPS`
5. 批量写入前后 `IO` 差异
6. 压缩对吞吐和 `IO` 的影响

建议测试场景：

1. 小 payload 高频写入
2. 大 payload 高频写入
3. 单线程与多线程对比
4. 不同 `batch_size`
5. 不同 `flush_interval`
6. 压缩开启与关闭对比

### 3.6 资源占用测试

目标：

1. 验证日志系统资源占用是否可接受
2. 验证长时间运行是否稳定

核心指标：

1. CPU 占用率
2. 内存占用
3. 队列峰值深度
4. 后台线程数
5. 文件句柄数量
6. 长稳运行后的磁盘占用变化

### 3.7 双接口一致性测试

目标：

1. 验证 `C++` 与 `Python` 只是入口不同，内部行为一致
2. 验证同样输入下产生日志格式一致

建议测试项：

1. 相同模块、相同级别、相同 payload 输出结果一致
2. 时间字段、模块字段、级别字段一致
3. 轮转、压缩、清理行为一致
4. 错误场景处理行为一致

## 4. 失败策略专项验证

失败策略是本次验证重点，需要单独设计专项场景。

### 4.1 队列满场景

验证内容：

1. `WARN` 以下是否允许按策略丢弃
2. `ERROR/CRITICAL` 是否仍尽量保留
3. 丢弃统计是否正确输出
4. 业务线程是否未被长时间阻塞

### 4.2 磁盘满场景

验证内容：

1. 是否进入降级写入模式
2. 是否按预期减少低优先级日志
3. 是否触发清理策略
4. 是否避免进程异常退出

### 4.3 压缩失败场景

验证内容：

1. 原文件是否仍保留
2. 是否输出内部错误日志
3. 后续新日志写入是否受影响

### 4.4 文件损坏场景

验证内容：

1. 启动时是否能识别坏文件
2. 坏文件是否被隔离或备份
3. 新文件是否能正常创建

### 4.5 异常退出场景

验证内容：

1. 最后少量日志丢失是否在可接受范围
2. 文件是否仍可读取
3. 重启后是否能继续写入

## 5. 性能评估指标

建议统一记录以下评估指标：

1. 单次接口平均耗时
2. 单次接口 `P95/P99` 耗时
3. 峰值吞吐量
4. 稳态吞吐量
5. 压缩前后磁盘写入量
6. CPU 平均占用与峰值占用
7. 内存平均占用与峰值占用
8. 发生降级前后的行为变化

## 6. 开源方案对比

### 6.1 对比对象

建议至少与以下方案对比：

1. `spdlog` 原生方案
2. `ROS` 常见日志方案
3. 现有 `naviai_log.hpp` 方案

### 6.2 对比维度

1. 对外接口易用性
2. 单次调用时延
3. 高并发吞吐量
4. 文件轮转与压缩能力
5. 异常恢复能力
6. 磁盘占用控制能力
7. 双语言接口支持能力
8. 多模块写入能力

### 6.3 对比输出

建议形成对比表，至少包含：

1. 功能支持项
2. 性能指标
3. 故障恢复表现
4. 接入复杂度
5. 适配本项目的优缺点

## 7. 测试输出物

测试完成后建议产出以下结果：

1. 测试用例清单
2. 测试脚本或压测程序
3. 原始测试数据
4. 关键指标统计表
5. 开源方案对比表
6. 问题列表与修复建议

## 8. 结论判定建议

建议按以下维度给出最终结论：

1. 功能是否通过
2. 失败策略是否通过
3. 实时性是否达标
4. 吞吐量是否达标
5. 磁盘 `IO` 是否可接受
6. 与开源方案相比是否具备落地价值

若上述任一关键项未通过，则不建议直接作为正式 `L3` 业务日志方案上线。
