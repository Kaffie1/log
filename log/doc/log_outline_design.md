# 日志模块概要设计

## 1. 文档目的

本文档用于描述日志模块的总体架构、核心模块划分、日志文件组织方式以及统一生命周期管理思路。

本文档定位为概要设计，重点回答：

1. 日志模块由哪些部分组成
2. 各部分之间如何协作
3. 日志文件如何组织
4. 日志生命周期由谁统一管理

## 2. 总体架构

日志模块整体由三类日志能力、一个通用日志封装模块、一个前台控制模块以及一个统一生命周期管理模块组成：

1. `L1` 系统观测日志模块
2. `L2` 原始数据日志模块
3. `L3` 业务日志 `SDK` 模块
4. `LogWriter` 通用日志封装模块
5. `LogService` 日志前台控制模块
6. `LogAgent` 日志生命周期管理模块

总体关系如下：

1. `L1` 和 `L3` 基于通用 `LogWriter` 写入文本日志
2. `LogWriter` 当前内部封装开源日志库 `spdlog`
3. `LogWriter` 与 `L1/L3` 保持解耦，后续如替换为其他开源日志库，仅调整 `LogWriter` 层即可
4. `L1/L3` 通过 `LogService` 获取目标文件、执行分段与封口控制
5. `L2` 通过专用 `RawData Writer` 写入原始数据和索引
6. 所有日志文件最终交由 `LogAgent` 统一扫描、压缩、清理、配额控制、打包、导出和上传

## 3. 模块划分

### 3.1 L1 系统观测日志模块

`L1` 负责记录系统运行状态和系统观测信息，主要包括：

1. `CPU / 内存 / 磁盘` 监控
2. 进程 / 线程 / `Docker` 监控
3. `GPU / 温度 / 网络` 监控
4. 系统事件与告警

`L1` 输出方式为标准文本日志，底层依赖通用 `LogWriter` 模块。

### 3.2 L2 原始数据日志模块

`L2` 负责记录原始数据，主要包括：

1. `Topic / TF / IMU / Odom`
2. `Laser / PointCloud / Image`
3. `CmdVel / 其他传感器数据`
4. 原始数据采集与落盘

`L2` 不依赖通用文本 `LogWriter` 写盘，而是通过内部 `RawData Writer` 独立完成原始数据和索引文件写入。

### 3.3 L3 业务日志 SDK 模块

`L3` 负责记录业务层日志，并通过对外 `SDK` 向业务模块提供统一接口。

典型接口包括：

1. `BIZ_INFO()`
2. `BIZ_WARN()`
3. `BIZ_ERROR()`

`L3` 底层同样基于通用 `LogWriter` 模块写入文本日志。

### 3.4 LogWriter 通用日志封装模块

`LogWriter` 是日志系统中的通用文本日志封装模块，负责对开源日志库进行统一适配。

当前实现可基于 `spdlog` 进行封装，但在架构上需要与 `L1`、`L3` 以及其他上层模块解耦。

其核心定位如下：

1. 对上提供统一日志写入接口
2. 对下屏蔽具体开源日志库实现差异
3. 当前适配 `spdlog`
4. 后续如替换为其他开源日志库，只需要调整 `LogWriter` 层，不影响上层模块调用方式

## 4. 写入模块设计

### 4.1 LogWriter 模块

`LogWriter` 是 `L1` 和 `L3` 共用的文本日志写入模块，也是日志系统对开源日志库的统一封装层。

当前 `LogWriter` 内部基于 `spdlog` 实现，但对上层不直接暴露 `spdlog` 细节。

其设计目标是：

1. 上层模块只依赖 `LogWriter` 抽象接口
2. 上层模块不直接依赖 `spdlog`
3. 若未来替换为其他日志库，只需修改 `LogWriter` 内部适配实现
4. `L1`、`L3` 的对外接口保持稳定

其核心能力包括：

1. `Level` 管理
2. `Pattern` 格式化
3. `Sink` 管理
4. `Async` 异步写入
5. `Rotate` 轮转
6. `Daily` 日志能力
7. `Flush` 策略
8. `ErrorHandler`

为了实现解耦，`LogWriter` 内部建议继续划分为两层：

1. 日志门面层  
   对外暴露统一接口，供 `L1` 和 `L3` 调用
2. 日志后端适配层  
   当前适配 `spdlog`，未来可替换为其他开源日志实现

### 4.3 LogService 模块

`LogService` 是日志系统中的前台控制模块，负责承接业务层的同步控制动作。

其核心能力包括：

1. 文件命名
2. 活跃文件创建
3. 分段控制
4. 文件封口
5. 取日志前强制切段

`LogService` 不负责后台扫描和治理，而是把已封口文件交给 `LogAgent` 继续处理。

### 4.2 RawData Writer 模块

`RawData Writer` 是 `L2` 内部原始数据写入器。

其核心能力包括：

1. `Data Writer`，负责写入 `.data` 文件
2. `Index Writer`，负责写入 `.idx` 文件
3. `Buffer` 管理
4. 文件分片 / 轮转
5. 数据校验
6. 异常恢复

设计上，`.data` 和 `.idx` 必须成对处理，保证原始数据和索引的一致性。

## 5. 日志文件目录结构

### 5.1 命名规则

所有日志文件统一遵循以下命名语义：

`开始时间-结束时间-后缀名`

其中：

1. `开始时间` 表示文件开始时间
2. `结束时间` 表示文件结束时间
3. 后缀名根据日志类型变化

示意格式：

```text
19700101_000000-19700101_235959-<suffix>
```

### 5.2 L1 日志文件

`L1` 系统观测日志目录如下：

```text
logs/l1/system/
```

文件示例：

```text
19700101_000000-19700101_235959.log
19700102_000000-19700102_235959.log
```

### 5.3 L2 日志文件

`L2` 原始数据日志目录如下：

```text
logs/l2/
```

其中：

1. `.data` 文件保存原始数据
2. `.idx` 文件保存索引
3. 两者按时间成对管理

文件示例：

```text
19700101_000000-19700101_235959.data
19700101_000000-19700101_235959.idx
```

### 5.4 L3 日志文件

`L3` 业务日志目录如下：

```text
logs/l3/navigation/
```

文件示例：

```text
19700101_000000-19700101_235959.log
19700102_000000-19700102_235959.log
```

## 6. LogAgent 生命周期管理设计

`LogAgent` 是统一管理所有日志文件生命周期的模块。

其职责不负责业务写入，也不负责前台命名和分段控制，而负责对已经生成的日志文件执行统一管理动作。

核心功能包括：

1. 扫描管理
2. 压缩管理
3. 清理管理
4. 配额控制
5. 打包管理
6. 导出管理
7. 上传管理

### 6.1 扫描管理

1. 定时扫描日志目录
2. 发现新增日志文件

### 6.2 压缩管理

1. 按策略压缩旧日志文件
2. 节省存储空间

### 6.3 清理管理

1. 按保留策略清理过期日志
2. 释放存储空间

### 6.4 配额控制

1. 监控磁盘配额
2. 防止日志写满磁盘

### 6.5 打包管理

1. 按任务或时间窗口生成日志包

### 6.6 导出管理

1. 导出到本地或外部存储
2. 支持导云端

### 6.7 上传管理

1. 上传到云端服务器
2. 支持断点续传

## 7. 数据流说明

日志系统的数据流如下：

1. `L1` 数据流向：`L1 -> LogService -> LogWriter -> LogAgent`
2. `L2` 数据流向：`L2 -> RawData Writer -> LogAgent`
3. `L3` 数据流向：`L3 -> LogService -> LogWriter -> LogAgent`
4. `LogWriter` 负责将 `L1/L3` 请求转换为统一文本日志写入行为
5. `LogService` 负责前台控制
6. `LogAgent` 负责所有日志文件的统一管理

其中：

1. `L1` 和 `L3` 共享通用文本日志写入路径
2. `L2` 走专用原始数据写入路径
3. 前台控制统一汇聚到 `LogService`
4. 生命周期管理统一汇聚到 `LogAgent`

## 8. 设计原则

本方案遵循以下设计原则：

1. `LogWriter` 仅负责文本日志的标准化写入
2. `LogWriter` 当前封装 `spdlog`，但必须与上层模块解耦
3. `L2` 使用内部 `RawData Writer`，不依赖 `LogWriter`
4. 所有日志文件统一命名规范，便于管理和检索
5. 日志文件的命名、分段、封口由 `LogService` 统一控制
6. 所有日志文件生命周期统一由 `LogAgent` 管理
6. `L2` 的 `.data` 与 `.idx` 文件必须成对处理

## 9. 存储去向

日志经过 `LogAgent` 统一管理后，最终可流向以下存储目标：

1. 长期存储
2. 备份存储
3. 云存储
4. 外部存储

## 10. 结论

日志模块概要架构可以概括为：

1. `L1` 负责系统观测日志
2. `L2` 负责原始数据日志
3. `L3` 负责业务日志 SDK
4. `LogWriter` 负责对 `spdlog` 等开源日志库的统一封装与文本日志写入
5. `RawData Writer` 负责原始数据与索引写入
6. `LogService` 负责前台文件控制
7. `LogAgent` 负责统一生命周期管理

该架构实现了日志能力分层、写入职责分离以及生命周期统一治理。
